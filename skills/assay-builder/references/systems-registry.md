# Systems registry

Layout v3+ introduces a per-workspace systems registry at `.assay/systems-registry.json` plus a contract file at `systems/<name>/system.yaml` for each registered system. This separates **system contracts** (managed by the framework) from **system source** (managed by the system itself, possibly in an independent git repository).

## Why contracts and not full templates

Before layout v3, the framework managed `systems/<core>/{README.md, framework.yaml, CHANGELOG.md, docs/*}` as templates. This conflicted with two real-world patterns:

- The active system was its own git repository with its own README/docs.
- A project superseded one system with another (e.g. a rewrite); the framework had no state machine to record the handover.

Layout v3 fixes both by recording each system as a registry entry with explicit `status`, `vcs`, `version`, and `supersedes`, and by tracking only the `system.yaml` contract — not the system's documentation or source.

## Registry schema

`.assay/systems-registry.json`:

```jsonc
{
  "__schema": 1,
  "primary": "active-system-name",      // exactly one system has status: primary
  "systems": {
    "active-system-name": {
      "name": "active-system-name",
      "path": "systems/active-system-name",
      "status": "primary",              // primary | active | superseded | archived
      "vcs": "independent-git",         // independent-git | embedded | none
      "vcs_ref": "main",
      "version": "1.0.0",
      "contract_file": "systems/active-system-name/system.yaml",
      "supersedes": ["older-system-name"],
      "absorbed_on": "2026-06-17",
      "archived_on": null,
      "archive_path": null
    },
    "older-system-name": { /* status: archived, archive_path: ... */ }
  },
  "updated_at": "..."
}
```

## Status state machine

| Status | Meaning | Transitions |
| --- | --- | --- |
| `primary` | The active main system. Exactly one allowed. | `system promote <other>` demotes this one to `superseded`. |
| `active` | Registered and live but not primary (e.g. an evaluator alongside a game). | `system promote` → `primary`. `system archive --apply` → `archived`. |
| `superseded` | Was primary; another system took over. | `system promote` → `primary`. `system archive --apply` → `archived`. |
| `archived` | Moved into `systems/archive/<date>-pre-<name>/`. Read-only. | Re-registering would require manual re-creation. |

`system archive` refuses to archive the current `primary`; promote a replacement first.

## VCS modes

- `independent-git` — system path is its own git repository with its own history. Root repo `.gitignore` should ignore the system path but allow the contract file:

  ```gitignore
  /systems/<name>/
  !/systems/<name>/system.yaml
  ```

  Framework `check` skips system internals; only `system.yaml` participates in manifest hashing.

- `embedded` — system source is tracked by the root repo. Useful for small frameworks where the system is an integral part of the repo.

- `none` — placeholder for systems that do not have version control yet (e.g. a brand-new directory). Treat as a TODO.

## Contract file (`system.yaml`)

Generated by `system register` (and by v2→v3 migration) at `systems/<name>/system.yaml`:

```yaml
system:
  project: <project-name>
  name: <system-name>
  version: 0.1.0
  status: primary
  vcs: independent-git
  vcs_ref: main
  supersedes: ["older-system-name"]
contract_managed_by: assay
```

The contract is the only system-side file the framework receives. System owners can edit it (e.g. bumping `version`), and `assay update` will treat it as `modified by user` and skip — never auto-overwrite.

## CLI

```bash
# Register an existing system directory in the registry
assay system register <path> \
  [--name <name>] \
  [--vcs independent-git|embedded|none] \
  [--vcs-ref <ref>] \
  [--system-version <semver>] \
  [--primary] \
  [--supersedes <name1,name2,...>]

# Promote another system to primary; demotes the previous one to superseded
assay system promote <selector>

# Archive a non-primary system (copy-first move into systems/archive/<date>-pre-<name>/)
assay system archive <selector> --dry-run
assay system archive <selector> --apply

# Inspect
assay system list [--status primary|active|superseded|archived] [--json]
assay system show <selector> [--json]
```

Selectors accept the full system name or a unique name prefix.

## Migration from layout v2

Run `assay migrate-layout --dry-run` first. The plan generates `create-systems-registry`, `generate-contract`, `mark-user-deleted`, and `upgrade-manifest` steps. After `--apply`:

1. The previous `manifest.project.core` becomes the `primary` system. Legacy `framework.yaml` (if present) is read for `status` and `supersedes` overrides.
2. Each `systems/*` directory gets a `system.yaml` contract; `.git` presence determines `independent-git` vs `embedded`.
3. Each `systems/archive/<date>/<name>` directory is registered as `archived`.
4. Old `systems/<core>/**` template entries are removed from `manifest.managed_files` and listed in `user_deleted`.

Run `assay check` afterwards. The previous error storm of "managed file missing" should be empty.
